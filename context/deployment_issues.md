## 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ (Resolved)

### DEP-001: Отсутствие файла deploy.sh на сервере

**Симптом:**
```
bash: ./scripts/deploy.sh: No such file or directory
```

**Корень проблемы:**
- Файл `deploy.sh` — это серверный скрипт, который **не коммитится в git** (содержит пути конкретного сервера).
- В репозитории есть только шаблон `deploy.sh.template`.
- При первом деплое шаблон не копируется в `deploy.sh` автоматически.

**Попытки решения (неудачные):**
1. `git checkout origin/main -- scripts/deploy.sh.template` — не сработало, т.к. локальные изменения на сервере (CSS, картинки) блокировали `git checkout`.
2. `curl -sfL https://raw.githubusercontent.com/.../deploy.sh.template` — не сработало, т.к. репозиторий может быть приватным или сервер не имеет доступа к raw.githubusercontent.com.

**Финальное решение:**
- **Полный скрипт деплоя встроен прямо в `.github/workflows/deploy.yml`** как SSH heredoc.
- Больше не нужны файлы на сервере — скрипт передаётся напрямую через SSH.

### DEP-002: `git checkout main` деплоил старый код

**Симптом:**
Код на сервере не обновлялся после пуша в `main`. Деплой "проходил", но сайт оставался на старой версии.

**Корень проблемы:**
`git checkout main` на сервере переключал на **локальную** ветку `main`, которая могла отставать от `origin/main` на десятки коммитов (из-за локальных изменений, которые блокировали `git pull`).

**Решение:**
Заменено на `git reset --hard origin/main` после `git fetch origin main`.

---

### DEP-003: Health check принимал "degraded" как успех

**Симптом:**
Деплой считался успешным, даже когда Redis был недоступен и приложение работало в деградированном режиме.

**Решение:**
Health check теперь принимает только `"healthy"`, отклоняя `"degraded"`.

---

### DEP-004: Rollback не восстанавливал схему БД

**Симптом:**
При откате код возвращался к старой версии, но база данных оставалась с новыми миграциями. Это вызывало несовместимость схемы.

**Решение:**
Rollback теперь восстанавливает БД из бэкапа, созданного **ДО** применения миграций (файлы с суффиксом `.pre-deploy`).

## 🟡 ПРОБЛЕМЫ СРЕДНЕЙ ВАЖНОСТИ (Partially Resolved)

### DEP-005: Локальные изменения на сервере блокируют git update

**Симптом:**
```
Your local changes to the following files would be overwritten by checkout:
  static/css/base.css
  static/img/flags/bel.png
  ...
```

**Корень проблемы:**
На сервере вручную редактировались CSS-файлы и заменены изображения флагов. `git reset --hard origin/main` **перезаписывает** эти изменения.

**Текущее состояние:**
`git reset --hard origin/main` работает, но **уничтожает все локальные правки** на сервере. Это компромисс: либо сервер всегда синхронизирован с репо, либо локальные правки сохраняются (но тогда код может отставать).

**Рекомендация на будущее:**
Все кастомизации (CSS, флаги) должны коммититься в репозиторий. Сервер — «cattle, not pet».

---

### DEP-006: Старые имена файлов флагов (bel.png → BLR.png)

**Симптом:**
Код ожидает `/static/img/flags/BLR.png`, но на сервере файл называется `bel.png`.

**Решение:**
Добавлен `LEGACY_FLAG_MAP` в `data/static_paths.py`, который автоматически маппит старые имена на новые:
- `bel.png` → `BLR.png`
- `rus.png` → `RUS.png`
- `kz.png` → `KAZ.png`
- `china.png` → `CHN.png`
- и т.д.

---

### DEP-007: Alembic не видит DATABASE_URL в non-interactive SSH

**Симптом:**
Alembic может использовать placeholder URL из `alembic.ini` вместо реального `DATABASE_URL`, так как SSH-сессия не загружает `.bashrc`.

**Решение:**
Перед запуском alembic скрипт делает:
```bash
set -a
source "$ENV_FILE"
set +a
DATABASE_URL="$DATABASE_URL" alembic upgrade head
```

## 🟢 НЕЗНАЧИТЕЛЬНЫЕ ПРОБЛЕМЫ (Monitoring)

### DEP-008: pip install --upgrade обновляет ВСЕ пакеты

`pip install -r requirements.txt --upgrade` может обновить зависимости до версий, которых нет в lock-файле. Это может привести к непредсказуемым изменениям поведения.

**Рекомендация:** Перейти на `pip-tools` или `uv` с lock-файлом.

### DEP-009: Нет блокировки параллельных деплоев

Если два пуша в `main` произойдут одновременно, два экземпляра скрипта запустятся параллельно. Это может вызвать race condition при миграциях БД.

**Рекомендация:** Добавить `flock` или проверку через GitHub Actions concurrency.

### DEP-010: SSL/TLS instability при git push с Windows

Периодические ошибки `schannel: failed to receive handshake, SSL/TLS connection failed` при пуше с Windows. Требуются повторные попытки.

## 📁 СТРУКТУРА ТЕКУЩЕГО ДЕПЛОЯ

```
.github/workflows/deploy.yml  ← Полный скрипт деплоя (embedded heredoc)
scripts/deploy.sh.template    ← Шаблон для ручного использования на сервере
scripts/deploy.sh             ← РЕАЛЬНЫЙ скрипт (создаётся на сервере вручную или из шаблона)
```

### Flow деплоя:
1. Push в `main` → запускается GitHub Actions
2. Job `test` → pytest с Redis, без rate limits
3. Job `deploy` → SSH на сервер → inline heredoc script:
   - `git reset --hard origin/main`
   - Бэкап БД → `backups/dev.db.YYYYMMDDHHMMSS.pre-deploy`
   - `pip install -r requirements.txt --upgrade`
   - `alembic upgrade head` (с DATABASE_URL из .env)
   - `systemctl restart shadow-hockey-league`
   - Health check (5 попыток, проверка JSON `"status": "healthy"`)
   - Автоматический rollback при ошибке

## 🔑 КЛЮЧЕВЫЕ УРОКИ

1. **Никогда не полагайся на файлы, которых нет в репозитории.** Всё, что нужно для деплоя, должно быть в git.
2. **`git checkout` ≠ `git reset --hard`.** Первый может быть no-op, второй — деструктивный, но гарантированный.
3. **Локальные изменения на сервере — зло.** Сервер должен быть «cattle, not pet».
4. **Встраивай скрипты в workflow YAML**, если они небольшие. Это устраняет целый класс проблем с «файл не найден».
5. **SSL push с Windows нестабилен.** Всегда проверяй `git ls-remote` после пуша.
6. **Legacy filenames** — добавляй маппинг в код, а не переименовывай файлы на проде.
