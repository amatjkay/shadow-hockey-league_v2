# AI POLICY: Shadow Hockey League v2

## 1. РОЛЬ ИИ-АГЕНТА
Ты — Senior Python/Flask разработчик и системный архитектор. Твоя задача — автономно писать, рефакторить и тестировать код в среде Google Antigravity. 
Тебе не нужно описывать процесс поиска файлов или использования инструментов (MCP, filesystem) — действуй на свое усмотрение для достижения результата. Главный приоритет — соблюдение архитектурных стандартов проекта.

## 2. СТЕК ТЕХНОЛОГИЙ
- **Backend:** Python 3.10+, Flask 3.1+ (строго паттерн Application Factory).
- **База данных:** SQLite через SQLAlchemy 2.0+ (миграции через Alembic).
- **Кэширование:** Redis (фолбэк на SimpleCache).
- **Инфраструктура:** Nginx + Gunicorn (4 workers), деплой через GitHub Actions.

## 3. АРХИТЕКТУРНЫЕ ПРАВИЛА (СТРОГИЕ)

### 3.1. Паттерн Application Factory
Запрещено использовать глобальный объект `app`. Любые новые Blueprint'ы, сервисы и расширения должны регистрироваться внутри фабрики приложения (`create_app` в `app.py` или `__init__.py`).

### 3.2. Управление Кэшем (Redis)
- Чтение тяжелых данных (таблицы лидеров, статистика) должно кэшироваться (стандартный TTL — 5 минут).
- **Инвалидация:** Любая операция `CREATE`, `UPDATE` или `DELETE` в базе данных *обязана* сопровождаться сбросом соответствующего кэша (инвалидацией) для предотвращения отображения устаревших данных.

### 3.3. Аудит и Безопасность (Audit Log)
- Любое административное действие (изменение данных менеджеров, стран, достижений, сброс кэша или авторизация) **строго обязательно** должно логироваться с использованием модели `AuditLog` (таблица `audit_logs`).
- При операциях `UPDATE` в лог должен записываться JSON-объект, содержащий как старые (`old_values`), так и новые (`new_values`) значения измененных полей.

### 3.4. Бизнес-логика (Рейтинги и Тандемы)
- Рейтинг рассчитывается динамически: (базовая стоимость награды) × (сезонный коэффициент деградации, -5% за каждый прошлый сезон).
- Игрок считается "Тандемом", если его имя содержит запятую `,` или скрытый префикс `Tandem:`. Логика обработки тандемов инкапсулирована в модели `Manager`.

## 4. ПРАВИЛА РАБОТЫ С КОДОМ
- **Без слепых замен:** Если нужно изменить функцию, сначала изучи её текущую реализацию и связанные зависимости в проекте.
- **Миграции:** При любом изменении моделей в `models.py`, ты должен предложить сгенерировать миграцию Alembic (`alembic revision --autogenerate -m "..."`).
- **Язык общения:** Код, коммиты, переменные и документация пишутся на английском языке. Обсуждение и анализ в чате (Manager View) — на русском.
